Typography
Each style token bundles five attributes: font family, size, weight, line height, and letter spacing. Use as complete units — never override individual attributes.

Default document flow uses h1–h4 only. For deeper content hierarchy, use a semantic text element + utility class instead (<p class="tw-text-title">, <label class="tw-text-label">, <span class="tw-text-caption">).
h5 is a limited exception for Upper labels (tags · status badges) and Contentful shifted mapping only. Do not use h6.

Contentful Shifted Header Mapping
In Contentful editing, Display occupies <h1>, so every other heading shifts down one level. Use this mapping only when authoring inside Contentful — never on standalone pages.

Why the shift? Contentful enforces a single global <h1> per page (the Display hero). Every editorial heading inherits one level deeper to preserve the document outline. Display is marketing hero only, one per page, and requires a CSS override in Contentful.

Migration Status (May 2026): The 2026 semantic utility classes ship today in @kiva/kv-tokens (verified via addUtilities in tailwind.config.js). The addBase element defaults are still on legacy styles (h1→textH1, h4→textH4, figcaption→textBase). Until addBase is migrated, apply the new semantic class explicitly when using h1–h4 (e.g. <h1 class="tw-text-headline">) so legacy auto-styles don't override.

Use the correct device tier
Mobile-first (no prefix) for XS+SM. tw-md: for tablet (734px+). tw-lg: for desktop (1024px+). Desktop and XL share the same token values — tw-lg: covers both.
Default all paragraphs to Base
tw-text-base is the default for all body text. Only override with a smaller token (caption, label) for a specific semantic purpose.
All Caps = 1–3 word labels only
tw-text-upper for tags and status badges only. Never apply to sentences, paragraphs, or anything longer than 3 words.
Never override token attributes individually
Do not override font-size, font-weight, line-height, or letter-spacing within a token. If nothing fits, open a request with the design system team.
Display = Marketing only, one per page
Not a global h1 default — requires a CSS override in Contentful. Never reuse on product pages, inside components, or in content cards.
Don't use Caption or Label for long paragraphs
These are UI label tokens. Long reading content always uses Base. Caption and Label are for metadata, form labels, filter names, and image captions.
These classes are still in the Tailwind config (addUtilities) for backwards compatibility. Prefer the 2026 semantic classes above for new work. Legacy classes are not marked deprecated but the semantic equivalents should be used going forward.

Layout rules
Align to column edges, not gutters
Gutters create breathing room between content blocks — they are not content starting points.
Use span to control content width
Content can span one or more columns. If span exceeds available columns, it wraps to the next row.
Intrinsic-width elements don't stretch
Tags, badges, pill buttons have a fixed natural size. Do not stretch or shrink them to fill the grid.
Fixed-width + auto content gaps
When a fixed-width element (e.g. side nav) sits next to auto content, use the parent gutter value for the gap.
Radius
8 tokens. All values from radius size.tokens.json and @kiva/kv-tokens/configs/tailwind.config.js. The base token (16px) maps to DEFAULT in Tailwind, generating tw-rounded (no suffix).
Click any radius name or tw-rounded-* class to copy
Concentric corners formula: inner radius = outer radius − gap
When placing an image or content block inside a card, always calculate the inner radius. Example: card (lg = 20px), gap/M (8px) → inner image = 12px (tw-rounded-md).
A 0px inner element inside a rounded card always looks like a layout error, not a design choice.

Borders live in the dedicated Border tab now. The 5 stroke tokens (solid/default · hover · active · selected · dashed) are documented there. This panel only covers corner-radius.
Match radius to element hierarchy
Cards/containers → lg/xl. Interactive controls (buttons, overlays) → base. Utility inputs → xs/sm. Radius should reflect visual prominence.
Mix tokens without hierarchical intent
Placing xl-radius cards next to base-radius buttons without a clear reason creates visual noise and undermines the token system.
Unify all overlays at base (16px)
Modal, lightbox, and bottom sheet all use base. This prevents radius-driven hierarchy between overlay types.
Use rounded-full on large surfaces
Reserve rounded-full for icon buttons, badges, and pill tags. Applying it to cards or panels looks unintentional and toy-like.
Elevation
3 state-based shadow tokens. Each is composed of Y offset, Blur, Spread, and Opacity, all on base color #000000. Values mirror Figma's canonical Drop shadow effect styles (Shadow Default / Hover / Click-Active). Source: Alias - Elevation library.
Click any token value to copy
State-based, not layer-based. Apply elevation/default to anything at rest (cards, inputs, search bars, dropdowns, modals — any z-layer). On hover, swap to elevation/hover. On press, swap to elevation/click-active. The shadow communicates interaction state, not z-depth.
Do not use tw-shadow or tw-shadow-lg for elevation. The Tailwind config values are: tw-shadow = 0 1px 4px 0 rgba(0,0,0,0.08) and tw-shadow-lg = 0 4px 12px rgba(0,0,0,0.08). Neither matches the elevation rule. Always apply elevation as custom box-shadow CSS from the token values shown below.

Swap shadows on state change, not z-layer

Best practices for applying border tokens consistently across components and themes.
Always reference the token, never hardcode
Use --border-default, --border-hover, etc. Never write border: 1px solid … with a raw pixel value. The token system depends on a single source of truth.
Border weight communicates state
1px = resting/hover/active (color signals the state) · 2px = selected/emphasis. Heavier always means more commitment. The former 1.2px step is retired.
Solid for filled content · dashed for empty
Style discipline. Solid borders signal "real content lives here." Dashed borders signal "slot waiting to be filled."
Don't use 2px or thicker on resting elements
2px is reserved for selected state. Applying it at rest robs the design system of its strongest state signal.
Don't signal selection with color change alone
Pair color change with the --border-selected 2px weight so selection is visible even when color contrast is low or color-blind users are affected.
Don't use dashed borders on filled content containers
Dashed implies "empty" or "incomplete." Applying it to a card with real content reads as a layout bug.
Don't reuse --border-* tokens to fake a focus ring
Focus uses a separate outline system. Stealing the border slot for focus changes the element's box size and breaks layout.
Use the legacy alias names if you must reference Figma
Designers may say "Stroke", "Outline", or "Border width" — all three refer to the same border token system in this library.
